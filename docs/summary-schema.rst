summary.json Schema
===================

Overview
--------

``summary.json`` is the compact, privacy-focused device summary generated by
``acdprobe`` collect mode.

Its purpose is to provide a stable, human-reviewable, machine-readable summary
of a monitor's USB HID control surface without requiring reviewers to inspect
the larger raw reports.

Typical usage::

    ./acdprobe --collect /dev/acdctl4
    ./acdprobe --profile basic-readonly /dev/acdctl4

The summary file is written to::

    probes/<VID_PID>/summary.json

Example::

    probes/05ac_9226/summary.json


Design Goals
------------

``summary.json`` is intended to be:

- compact
- deterministic
- privacy-focused by default
- easy to diff in Git
- easy to aggregate into a future device database

It is a summary of probe results, not a complete raw dump. The complete raw
information remains available in:

- ``report.json``
- ``report.txt``
- ``report.csv``
- ``report_descriptor.hex``


Privacy Model
-------------

``summary.json`` is intended to be shareable.

It omits host-specific topology or path information such as:

- full sysfs paths
- USB physical topology
- bus/device runtime numbering
- USB serial strings
- usernames
- hostnames
- time zone data
- timestamps in filenames

The retained USB/sysfs identity fields are limited to device-level comparison
signals such as manufacturer, product, vendor ID, product ID, and
``bcdDevice``.


Schema Stability
----------------

The schema is versioned by ``schema_version``.

Rules:

- New fields may be added in future schema versions.
- Existing field meanings should remain stable whenever possible.
- Unknown fields should be ignored by consumers.
- Contributors and reviewers should prefer ``summary.json`` for quick review,
  and fall back to the larger raw files when needed.


Current Top-Level Fields
------------------------

``schema_version``
    Integer schema version for ``summary.json``.

``tool``
    Object describing the tool that generated the file.

``device``
    Object describing the probed HID device identity.

``applications``
    Array of top-level HID application usages reported by the device.

``reports``
    Compact count summary for input/output/feature reports.

``report_descriptor``
    Compact fingerprint of the HID report descriptor.

``controls``
    Array of normalized control summaries.

``telemetry_candidates``
    Array of unresolved vendor-private controls summarized for reverse
    engineering.


Current Field Reference
-----------------------

``schema_version``
~~~~~~~~~~~~~~~~~~

Type:
    integer

Meaning:
    Version of the ``summary.json`` schema.

Example::

    "schema_version": 1


``tool``
~~~~~~~~

Type:
    object

Current fields:

``name``
    Tool name.

``version``
    Tool version string.

Example::

    "tool": {
      "name": "acdprobe",
      "version": "0.5"
    }


``device``
~~~~~~~~~~

Type:
    object

Current fields:

``vid_pid_key``
    Stable key in ``VVVV_PPPP`` form, derived from USB vendor ID and product ID.

``vendor``
    USB vendor ID as hexadecimal string.

``product``
    USB product ID as hexadecimal string.

``version``
    USB device version as hexadecimal string.

``ifnum``
    USB interface number used for the HID probe.

``hid_name``
    HID device name, if available.

``manufacturer_string``
    USB manufacturer string, if available.

``product_string``
    USB product string, if available.

``usb_sysfs``
    Privacy-filtered USB identity values gathered from sysfs when available.
    Current fields are:

    - ``manufacturer``
    - ``product``
    - ``idVendor``
    - ``idProduct``
    - ``bcdDevice``

Example::

    "device": {
      "vid_pid_key": "05ac_9226",
      "vendor": "0x05ac",
      "product": "0x9226",
      "version": "0x0299",
      "ifnum": 0,
      "hid_name": "Apple Inc. Apple LED Cinema Display",
      "manufacturer_string": "Apple Inc.",
      "product_string": "Apple LED Cinema Display",
      "usb_sysfs": {
        "manufacturer": "Apple Inc.",
        "product": "Apple LED Cinema Display",
        "idVendor": "05ac",
        "idProduct": "9226",
        "bcdDevice": "0299"
      }
    }


``applications``
~~~~~~~~~~~~~~~~

Type:
    array of objects

Each entry describes a HID application usage reported by the device.

Current fields:

``usage_code``
    Full HID usage code as hexadecimal string.

``decoded``
    Human-readable decoded name.

Example::

    "applications": [
      {
        "usage_code": "0x00800001",
        "decoded": "Monitor Control"
      }
    ]


``reports``
~~~~~~~~~~~

Type:
    object

Current fields:

``input_count``
    Number of input reports discovered.

``output_count``
    Number of output reports discovered.

``feature_count``
    Number of feature reports discovered.

Example::

    "reports": {
      "input_count": 1,
      "output_count": 0,
      "feature_count": 4
    }


``report_descriptor``
~~~~~~~~~~~~~~~~~~~~~

Type:
    object

Current fields:

``available``
    True when a report descriptor was successfully read.

``byte_count``
    Descriptor size in bytes.

``fnv1a64``
    Stable 64-bit FNV-1a fingerprint of the raw descriptor bytes, formatted as a
    hexadecimal string.

``preview_hex``
    First 16 descriptor bytes rendered as space-separated hexadecimal values.

Example::

    "report_descriptor": {
      "available": true,
      "byte_count": 78,
      "fnv1a64": "0xf2f86cccedaaf3cf",
      "preview_hex": "05 80 09 01 a1 01 75 10 95 01 15 00 26 ff 03 06"
    }


``controls``
~~~~~~~~~~~~

Type:
    array of objects

Each entry is a normalized control summary.

Current fields:

``confidence``
    Confidence label for the interpretation of the control.

``report_type``
    HID report type such as ``input`` or ``feature``.

``report_id``
    HID report ID containing the control.

``field_index``
    HID field index within the report.

``application_usage``
    HID application usage code associated with the field.

``application_decoded``
    Human-readable application name.

``usage_code``
    Full HID usage code as hexadecimal string.

``usage_page``
    HID usage page extracted from ``usage_code``.

``usage_decoded``
    Human-readable control name.

``usage_count``
    Number of values exposed by the field for this usage.

``logical``
    Raw HID field logical value rendered as hexadecimal string.

``physical``
    Raw HID field physical value rendered as hexadecimal string.

``logical_minimum``
    Declared HID logical minimum.

``logical_maximum``
    Declared HID logical maximum.

``physical_minimum``
    Declared HID physical minimum.

``physical_maximum``
    Declared HID physical maximum.

``field_flags``
    Numeric HID field flags bitmask.

``field_flags_text``
    Human-readable field flags summary.

``unit_exponent``
    HID unit exponent.

``unit``
    HID unit value.

``is_single_value``
    True when the current value is scalar rather than an array.

``value_count``
    Number of current values represented.

``current_value``
    Current observed value. This may be a number, string, or array of values.

Example::

``telemetry_candidates``
~~~~~~~~~~~~~~~~~~~~~~~~

Type:
    array of objects

Each entry summarizes an unresolved vendor-private control for quicker reverse
engineering review.

Current fields:

``report_type``
    HID report type such as ``feature``.

``report_id``
    HID report ID containing the candidate.

``field_index``
    HID field index within the report.

``usage_code``
    Full HID usage code as hexadecimal string.

``usage_page``
    HID usage page extracted from ``usage_code``.

``usage_decoded``
    Human-readable decoded usage name.

``confidence``
    Confidence label for the current interpretation.

``logical_minimum``
    Declared HID logical minimum.

``logical_maximum``
    Declared HID logical maximum.

``field_flags``
    Numeric HID field flags bitmask.

``field_flags_text``
    Human-readable field flags summary.

``readable``
    True when a current value was successfully read.

``looks_like``
    Coarse shape hint such as ``boolean``, ``enum``, ``scalar``, ``blob-like``,
    or ``unreadable``.

``current_value``
    Current observed value. This may be a number or an array of values.

        "readable": true,
        "looks_like": "boolean",
        "current_value": 0
      },
      {
        "report_type": "feature",
        "report_id": 236,
        "field_index": 1,
        "usage_code": "0xff9200ec",
        "usage_page": "0xff92",

        "logical_minimum": -2147483648,

      }

Confidence Levels
-----------------

``known``
~~~~~~~~~

Use when the control meaning is supported by strong evidence, such as:

- public specification
- vendor documentation
- repeatable controlled testing with clear interpretation
- prior established support in the project

Examples:

- brightness on Apple LED Cinema Display 27-inch
- ambient light sensor / auto-brightness control mapped to VESA/MCCS 0x66


``candidate``
~~~~~~~~~~~~~

Use when a control has a plausible partial interpretation that is already useful
for investigation, but is not yet strong enough to be treated as established.

Typical reasons:

- vendor-private control with a plausible type or role
- controlled testing ruled out some interpretations but did not confirm one
- the current name remains intentionally provisional

Example:

- vendor-private boolean fields that are under active testing


``tentative``
~~~~~~~~~~~~~

Use when a control is real and worth surfacing, but its meaning is still only
partially understood.

Typical reasons:

- vendor-private data/status fields under investigation
- behavior observed in testing but not yet fully characterized
- meaning inferred from patterns but not yet proven


``observed``
~~~~~~~~~~~~

Use when a control is present and recorded, but the project is not yet making a
stronger interpretation claim.

This label is appropriate for preserving visibility without overstating meaning.


``unknown``
~~~~~~~~~~~

Use when a control exists but current code cannot assign a more specific label.

This label should be uncommon in curated summaries, but consumers should be able
to handle it.


Confidence Guidelines
---------------------

When in doubt:

- prefer ``candidate`` or ``tentative`` over overconfident claims
- prefer ``observed`` over a guess that looks more precise than the evidence
- reserve ``known`` for mappings supported by strong evidence

The goal is to keep the device database trustworthy.

A later commit or future research may promote a control from:

- ``observed`` -> ``candidate``
- ``candidate`` -> ``tentative``
- ``tentative`` -> ``known``


Contributor Guidance
--------------------

For community submissions, preferred commands are::

    ./acdprobe --profile basic-readonly /dev/acdctl4

or::

    ./acdprobe --collect /dev/acdctl4

These commands avoid experimental writes and produce privacy-focused output.

Experimental workflows such as::

    ./acdprobe --profile als-discovery ...

should only be used when intentionally helping reverse-engineer monitor
features.


Reviewer Guidance
-----------------

When reviewing a contributed probe bundle:

1. Start with ``summary.json``.
2. Check whether the VID:PID, HID name, ``bcdDevice``, and report descriptor
   fingerprint match an existing known device.
3. Review ``controls`` and confidence labels.
4. Review ``telemetry_candidates`` for unresolved vendor-private fields.
5. Use ``report.json`` and ``report_descriptor.hex`` only when deeper inspection
   is needed.


Notes
-----

``summary.json`` is a project interface for data collection and review.
It should remain conservative, explicit, easy to compare across devices, and
safe to share by default.
