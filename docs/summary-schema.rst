summary.json Schema
===================

Overview
--------

``summary.json`` is the compact device summary generated by ``acdprobe``
collect mode.

Its purpose is to provide a stable, human-reviewable, machine-readable summary
of a monitor's USB HID control surface without requiring reviewers to inspect
all of the larger raw reports first.

Typical usage::

    ./acdprobe --collect /dev/acdctl4
    ./acdprobe --profile basic-readonly /dev/acdctl4

The summary file is written to::

    probes/<VID_PID>/summary.json

Example::

    probes/05ac_9226/summary.json


Design Goals
------------

``summary.json`` is intended to be:

- compact
- deterministic
- easy to diff in Git
- easy to aggregate into a future device database
- descriptive enough for cross-machine and cross-platform comparison

It is a summary of probe results, not a complete raw dump. The complete raw
information remains available in:

- ``report.json``
- ``report.txt``
- ``report.csv``
- ``report_descriptor.hex``


Privacy Notes
-------------

``summary.json`` is intended to be shareable for community submissions, but it
is no longer a minimal privacy-only view.

The current schema includes:

- USB identity strings
- USB revision-like fields from sysfs
- a resolved USB device sysfs path
- USB runtime bus/device numbering
- HID application and control metadata

In current output, the ``device.usb_sysfs`` object includes:

- ``device_realpath``
- ``manufacturer``
- ``product``
- ``serial``
- ``idVendor``
- ``idProduct``
- ``bcdDevice``
- ``busnum``
- ``devnum``
- ``speed``

Contributors should review ``summary.json`` before sharing it if they are
concerned about exposing local USB topology or runtime enumeration details.


Schema Stability
----------------

The schema is versioned by ``schema_version``.

Rules:

- New fields may be added in future schema versions.
- Existing field meanings should remain stable whenever possible.
- Unknown fields should be ignored by consumers.
- Contributors and reviewers should prefer ``summary.json`` for quick review,
  and fall back to the larger raw files when needed.


Current Top-Level Fields
------------------------

``schema_version``
    Integer schema version for ``summary.json``.

``tool``
    Object describing the tool that generated the file.

``device``
    Object describing the probed HID device identity and collected USB sysfs
    identity fields.

``applications``
    Array of top-level HID application usages reported by the device.

``reports``
    Compact count summary for input/output/feature reports.

``report_descriptor``
    Compact fingerprint and preview of the HID report descriptor.

``telemetry_candidates``
    Array of condensed summaries for unresolved vendor-private controls that
    are useful for reverse-engineering and cross-device comparison.

``controls``
    Array of normalized control summaries, including known, candidate,
    tentative, and observed controls.


Current Field Reference
-----------------------

``schema_version``
~~~~~~~~~~~~~~~~~~

Type:
    integer

Meaning:
    Version of the ``summary.json`` schema.

Example::

    "schema_version": 1


``tool``
~~~~~~~~

Type:
    object

Current fields:

``name``
    Tool name.

``version``
    Tool version string.

Example::

    "tool": {
      "name": "acdprobe",
      "version": "0.5"
    }


``device``
~~~~~~~~~~

Type:
    object

Current fields:

``vid_pid_key``
    Stable key in ``VVVV_PPPP`` form, derived from USB vendor ID and product ID.

``vendor``
    USB vendor ID as hexadecimal string.

``product``
    USB product ID as hexadecimal string.

``version``
    USB device version as hexadecimal string.

``ifnum``
    USB interface number used for the HID probe.

``hid_name``
    HID device name, if available.

``manufacturer_string``
    USB manufacturer string, if available.

``product_string``
    USB product string, if available.

``usb_sysfs``
    Object containing USB identity fields harvested from sysfs when available.

``device.usb_sysfs`` current fields:

``device_realpath``
    Resolved sysfs path of the parent USB device.

``manufacturer``
    USB manufacturer string read from sysfs.

``product``
    USB product string read from sysfs.

``serial``
    USB serial string read from sysfs, or ``null`` when unavailable.

``idVendor``
    USB vendor ID from sysfs, without ``0x`` prefix.

``idProduct``
    USB product ID from sysfs, without ``0x`` prefix.

``bcdDevice``
    USB device release / revision field from sysfs.

``busnum``
    USB bus number as reported by sysfs.

``devnum``
    USB device number as reported by sysfs.

``speed``
    USB speed string as reported by sysfs.

Example::

    "device": {
      "vid_pid_key": "05ac_9226",
      "vendor": "0x05ac",
      "product": "0x9226",
      "version": "0x0299",
      "ifnum": 0,
      "hid_name": "Apple Inc. Apple LED Cinema Display",
      "manufacturer_string": "Apple Inc.",
      "product_string": "Apple LED Cinema Display",
      "usb_sysfs": {
        "device_realpath": "/sys/devices/.../usb1/1-2/1-2.6",
        "manufacturer": "Apple Inc.",
        "product": "Apple LED Cinema Display",
        "serial": null,
        "idVendor": "05ac",
        "idProduct": "9226",
        "bcdDevice": "0299",
        "busnum": "1",
        "devnum": "19",
        "speed": "12"
      }
    }


``applications``
~~~~~~~~~~~~~~~~

Type:
    array of objects

Each entry describes a HID application usage reported by the device.

Current fields:

``usage_code``
    Full HID usage code as hexadecimal string.

``decoded``
    Human-readable decoded name.

Example::

    "applications": [
      {
        "usage_code": "0x00800001",
        "decoded": "Monitor Control"
      }
    ]


``reports``
~~~~~~~~~~~

Type:
    object

Current fields:

``input_count``
    Number of input reports discovered.

``output_count``
    Number of output reports discovered.

``feature_count``
    Number of feature reports discovered.

Example::

    "reports": {
      "input_count": 1,
      "output_count": 0,
      "feature_count": 4
    }


``report_descriptor``
~~~~~~~~~~~~~~~~~~~~~

Type:
    object

Current fields:

``available``
    Boolean indicating whether the report descriptor was found and read.

``byte_count``
    Descriptor size in bytes.

``fnv1a64``
    FNV-1a 64-bit fingerprint of the raw descriptor bytes, rendered as a
    hexadecimal string.

``preview_hex``
    Hex preview of the first 16 descriptor bytes.

Example::

    "report_descriptor": {
      "available": true,
      "byte_count": 78,
      "fnv1a64": "0xf2f86cccedaaf3cf",
      "preview_hex": "05 80 09 01 a1 01 75 10 95 01 15 00 26 ff 03 06"
    }


``telemetry_candidates``
~~~~~~~~~~~~~~~~~~~~~~~~

Type:
    array of objects

Purpose:
    Surface unresolved vendor-private controls in a compact form so reviewers
    can compare likely telemetry or status fields without reading the full raw
    report dump first.

Selection rule:
    The current implementation includes unresolved vendor-private controls that
    are still under investigation and omits already-known public controls such
    as brightness and the standard ambient-light-sensor control.

Current fields:

``report_type``
    Report type string, currently typically ``feature``.

``report_id``
    HID report ID containing the candidate control.

``field_index``
    Field index within the report.

``usage_code``
    Full HID usage code as hexadecimal string.

``usage_page``
    HID usage page as hexadecimal string.

``usage_decoded``
    Human-readable decoded usage description.

``confidence``
    Reverse-engineering confidence label for the candidate.

``logical_minimum``
    HID logical minimum for the field.

``logical_maximum``
    HID logical maximum for the field.

``field_flags``
    Raw HID field flags as integer.

``field_flags_text``
    Human-readable field flags summary.

``readable``
    Boolean indicating whether the current value could be read successfully.

``looks_like``
    Coarse shape classification derived from the field metadata and current
    value shape. Current values include ``boolean``, ``enum``, ``scalar``,
    ``blob-like``, and ``unreadable``.

``current_value``
    Current value, either a scalar integer or an array of integers depending on
    the field shape.

Example::

    "telemetry_candidates": [
      {
        "report_type": "feature",
        "report_id": 225,
        "field_index": 0,
        "usage_code": "0xff9200e1",
        "usage_page": "0xff92",
        "usage_decoded": "Vendor-private boolean (tentative)",
        "confidence": "candidate",
        "logical_minimum": 0,
        "logical_maximum": 1,
        "field_flags": 2,
        "field_flags_text": "variable",
        "readable": true,
        "looks_like": "boolean",
        "current_value": 0
      },
      {
        "report_type": "feature",
        "report_id": 236,
        "field_index": 1,
        "usage_code": "0xff9200ec",
        "usage_page": "0xff92",
        "usage_decoded": "Vendor-private data/status (tentative)",
        "confidence": "tentative",
        "logical_minimum": -2147483648,
        "logical_maximum": 2147483647,
        "field_flags": 2,
        "field_flags_text": "variable",
        "readable": true,
        "looks_like": "blob-like",
        "current_value": [16252, 64473]
      }
    ]


``controls``
~~~~~~~~~~~~

Type:
    array of objects

Each entry is a normalized control summary.

Current fields:

``report_type``
    HID report type containing the control, such as ``feature`` or ``input``.

``report_id``
    HID report ID containing the control.

``field_index``
    HID field index containing the control.

``application_usage``
    HID application usage code for the field, as hexadecimal string.

``application_decoded``
    Human-readable decoded application usage.

``usage_code``
    Full HID usage code as hexadecimal string.

``usage_page``
    HID usage page extracted from ``usage_code``, as hexadecimal string.

``usage_decoded``
    Human-readable decoded usage name.

``name``
    Project-facing display name for the control.

``confidence``
    Confidence label for the control interpretation.

``logical``
    Field logical value, rendered as hexadecimal string.

``physical``
    Field physical value, rendered as hexadecimal string.

``logical_minimum``
    HID logical minimum for the field.

``logical_maximum``
    HID logical maximum for the field.

``physical_minimum``
    HID physical minimum for the field.

``physical_maximum``
    HID physical maximum for the field.

``field_flags``
    Raw HID field flags as integer.

``field_flags_text``
    Human-readable field flags label.

``unit_exponent``
    HID unit exponent.

``unit``
    HID unit value.

``usage_count``
    Number of usages carried by the field entry represented by this control.

``is_single_value``
    Boolean indicating whether ``current_value`` is a scalar rather than an
    array.

``value_count``
    Number of values represented in ``current_value``.

``current_value``
    Current observed value. This may be:

    - a number
    - a string
    - an array of values

Confidence Levels
-----------------

Current confidence labels are:

``known``
~~~~~~~~~

Use when the control meaning is supported by strong evidence, such as:

- public specification
- vendor documentation
- repeatable controlled testing with clear interpretation
- prior established support in the project

Examples:

- brightness on Apple LED Cinema Display 27-inch
- ambient light sensor / auto-brightness control mapped to VESA/MCCS 0x66


``candidate``
~~~~~~~~~~~~~

Use when the interpretation is plausible and useful, but not yet strong enough
to be treated as established.

Typical reasons:

- a vendor-private control that appears important
- partial reverse-engineering with some controlled tests
- evidence suggests a role, but the exact meaning remains incomplete

Example:

- vendor-private boolean fields under active investigation


``tentative``
~~~~~~~~~~~~~

Use when the control is recorded and partially described, but its role is still
under investigation.

Typical reasons:

- vendor-private status/data fields with unstable or unclear semantics
- fields that are visible and worth tracking but not yet confidently mapped

Example:

- vendor-private data/status fields under investigation


``observed``
~~~~~~~~~~~~

Use when a control is present and described from the HID metadata, but the
project is not yet making a stronger claim about its meaning.


``unknown``
~~~~~~~~~~~

Use when a control is present and recorded, but even a basic interpretation is
not yet available.


Confidence Guidelines
---------------------

When in doubt:

- prefer ``candidate`` or ``tentative`` over ``known``
- prefer ``observed`` or ``unknown`` over an overconfident guess

The goal is to keep the device database trustworthy.

A later commit or future research may promote a control from:

- ``unknown`` -> ``observed``
- ``observed`` -> ``candidate``
- ``candidate`` -> ``tentative``
- ``tentative`` -> ``known``

The exact path does not have to be linear; use the smallest label that matches
current evidence.


Contributor Guidance
--------------------

For community submissions, preferred commands are::

    ./acdprobe --profile basic-readonly /dev/acdctl4

or::

    ./acdprobe --collect /dev/acdctl4

These commands avoid experimental writes and produce a stable summary plus the
raw supporting files. Review the generated files before sharing them if USB
topology disclosure is sensitive in your environment.

Experimental workflows such as::

    ./acdprobe --profile als-discovery ...

should only be used when intentionally helping reverse-engineer monitor
features.


Reviewer Guidance
-----------------

When reviewing a contributed probe bundle:

1. Start with ``summary.json``.
2. Check whether the VID:PID, USB revision fields, and HID name match an
   existing known device.
3. Check the ``report_descriptor`` fingerprint for a quick revision signature.
4. Review ``telemetry_candidates`` for unresolved vendor-private controls.
5. Review ``controls`` and confidence labels.
6. Use ``report.json`` and ``report_descriptor.hex`` only when deeper inspection
   is needed.


Notes
-----

``summary.json`` is a project interface for data collection and review.
It should remain conservative, explicit, and easy to compare across devices.
